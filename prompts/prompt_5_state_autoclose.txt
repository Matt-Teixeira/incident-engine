Phase 5 - State Machine + Auto-Close (L5)

!! REVISED 2026-07-16 (FLOW Step 3, pre-implementation) - see
!! notes/phase_5_reevaluation.md for the live evidence and the developer decisions.
!! The phase's SHAPE is unchanged (pure domain/state.js transitions, oracle SELECT-only,
!! engine-driven transitions only, re-open on post-resolve recurrence, acknowledged/
!! suppressed defined-but-never-set, no L4/notifications/LLM). Five ORIGINAL clauses
!! were wrong or missing against the live system and are SUPERSEDED by the Scope below:
!!   - "resolvable systems[]": DEAD - systems[] is structurally <=1 (the entity IS the
!!     system; empty for __global__). Auto-close joins the oracle on entity.
!!   - the recurring trigger ("threshold or multi-run span"): would mark ~ALL incidents
!!     recurring instantly (303/504 span >=9 days; everything recurs every cron cycle).
!!     recurring now means exactly: re-opened after a resolution.
!!   - the oracle can only ever speak to 60% of incidents (188 on entities it has never
!!     recorded + 11 __global__). A second deterministic close was added:
!!     resolved_reason='stale' (no recurrence in STALE_AFTER_DAYS, default 7).
!!   - "capture_datetime vs inserted_at" is RESOLVED: capture_datetime (0 nulls across
!!     89k rows; inserted_at mis-orders backfills by up to 76 days).
!!   - re-open compared producer-clock last_seen against DB-clock resolved_at - producer
!!     skew could leave an incident resolved WHILE FAILING. New column
!!     resolved_last_seen makes re-open a same-clock-domain comparison.

Goal:
Give incidents a deterministic lifecycle and auto-resolve them when the underlying system
recovers (stats.acquisition_history.successful_acquisition as the recovery oracle) or when
the problem stops occurring (staleness) - so the incident list reflects current reality
without a human closing each one.

Scope:
- State vocabulary on incidents.incidents.state: open / acknowledged / recurring /
  resolved / suppressed. ENGINE-DRIVEN TRANSITIONS ONLY:
  - initialization: a new (fingerprint, entity) -> open. The existing backlog (state IS
    NULL, ~504 rows live) -> open on the first run (Phase 4 backlog precedent).
  - ordinary re-occurrence while open/recurring: NO state change - occurrence_count/
    last_seen already advance via the aggregate. (The original threshold/span ->
    recurring trigger is superseded: it is crossed by ~everything instantly, and a state
    everything has distinguishes nothing - the Phase 4 flat-medium lesson.)
  - auto-close (recovery): for each open/recurring/acknowledged incident whose entity is a
    system the oracle records: if stats.acquisition_history has successful_acquisition=true
    for system_id = entity with capture_datetime LATER than the incident's last_seen ->
    state=resolved, resolved_reason='auto_recovery'. capture_datetime, NOT inserted_at
    (same producer-side clock domain as the event dt feeding last_seen; inserted_at lags
    it by a median ~5min but up to 76 days on backfills, which would close incidents on
    stale evidence). COALESCE(capture_datetime, inserted_at) only as the documented
    fallback for the column's nominal nullability (0 nulls live).
  - auto-close (staleness): any incident (open/recurring/acknowledged) with NO recurrence
    in STALE_AFTER_DAYS -> state=resolved, resolved_reason='stale'. STALE_AFTER_DAYS is a
    named constant, default 7 - above the live quiet-1-3d hump (77 incidents merely
    between recurrences must not close) and at the source-retention horizon; re-justify
    against live recency in Step 2. This is what lets the ~40% of incidents the oracle
    cannot see (__global__ + oracle-absent entities - live: 188+11 of 504) ever resolve.
    PRECEDENCE: recovery is evaluated first, staleness second, so an incident eligible
    for both records the stronger reason.
  - on ANY resolve: stamp resolved_at (the batch's evaluation time, supplied by the job)
    AND resolved_last_seen = the incident's last_seen at that moment. resolved_at is
    stamped once per transition, never refreshed by re-runs.
  - re-open: a resolved incident with last_seen > resolved_last_seen -> recurring
    (NOT plain open - recurring's one meaning is "came back after we closed it"; it is
    also the flap signal: 60 live entities interleave successes and failures within 48h,
    so close->re-open cycling WILL happen and must be visible). The comparison is
    last_seen vs resolved_last_seen - both producer-clock values - NEVER vs resolved_at
    (DB clock): a producer whose clock trails the DB could otherwise advance last_seen
    without ever exceeding resolved_at, leaving the incident resolved while failing.
    Clear resolved_* on re-open? NO - keep them as history of the last resolution; the
    state field says it's back.
  - acknowledged and suppressed are reserved for a future human/dashboard action - the
    engine defines but NEVER SETS them. The transition table must still define behavior
    FROM them: acknowledged is auto-closeable (recovery and staleness both apply - ack
    means "a human saw it", not "keep it open"); suppressed is ENGINE-TERMINAL (no engine
    transition out, ever - only a human un-suppresses).
- db/schema.sql: add resolved_last_seen TIMESTAMPTZ to incidents.incidents (CREATE + an
  idempotent Phase 5 UPGRADE section - mirror the entity/type precedents). Update
  docs/incidents-schema.md to match.
- jobs/assess (state step): runs AFTER assessment in the same job. Evaluates ALL incidents
  from durable facts every run (the Phase 4 sibling-staleness lesson - no touched-only
  predicate; no watermark: re-runnability is a property of recomputing from durable facts),
  writes only rows whose state actually changed (no-op filter; unchanged rows must not
  churn updated_at every cron cycle).
- domain/state.js: the PURE transition function - (incident facts + recovery facts +
  evaluation time) -> {state, resolved_reason?} or no-change. NO clock inside (the job
  supplies the evaluation timestamp), no DB handle, no network. The original prompt's
  "resolved_at=now()" wording is superseded: the pure function never calls now().
- utils/db/queries/recovery.js: the oracle lookup (max successful capture_datetime per
  entity for this batch's incident entities). SELECT-only. 89k rows live - bound sanely
  and confirm the plan in Step 2.

Non-goals:
- NO auto-remediation (L4): action_state/action_ref exist but are never written.
- No notifications. No human-transition UI. No writes outside incidents. No LLM.
- No flap dampening/hysteresis: this phase makes flapping VISIBLE (recurring), not
  suppressed. If churn proves noisy, hysteresis is a future, logged decision.
- severity is NOT a state input - lifecycle and severity stay orthogonal this phase.
- The queued classification phase (Phase 4 M2 follow-up) runs AFTER this phase
  (developer-decided 2026-07-16) and keeps its obligations (interim-M2 reason string,
  RULES_VERSION bump).
